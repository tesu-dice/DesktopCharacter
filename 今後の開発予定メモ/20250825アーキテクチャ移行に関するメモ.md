# EventBusアーキテクチャへの移行計画書

このドキュメントは、アプリケーションの構造をEventBusを用いた疎結合なアーキテクチャへ移行するための、最終的な方針と具体的な手順を定めたものです。

---

### 1. 現状のアーキテクチャ（移行前）

現在のアプリケーションは、以下の特徴を持つ「密結合」な構造になっています。

-   **インスタンス管理:**
    -   `main.py`が全ての機能（サービス）のインスタンスを生成し、管理しています。
-   **情報伝達:**
    -   設定情報は、各サービスの初期化時に引数として直接渡されます。
    -   サービス間の連携は、`main.py`を介したり、サービス同士が互いの関数を直接呼び出すことで実現されています。
-   **共通機能:**
    -   音声ライブラリのリスト取得など、一部の共通機能がモジュール内のグローバル関数として実装されています。

---

### 2. 新アーキテクチャの基本方針（移行後）

移行後は、**「AppContext & EventBus (集中管理型)」** パターンを正式に採用します。

-   **司令塔 (AppContext):**
    -   `main.py`内の`myapp`クラスが、引き続き全サービスのインスタンスを管理する「司令塔」の役割を担います。
-   **郵便局 (EventBus):**
    -   サービス間の直接呼び出しを全廃します。
    -   代わりに`EventBus`のインスタンスを介した「イベント」のやり取り（発行/購読）で、非同期に連携します。
-   **配線の集中管理:**
    -   **【重要ルール】** イベントの購読（`subscribe`）設定は、**全て`main.py`の`myapp`クラス内で一元的に行います。** これにより、アプリケーション全体の動作の見通しを確保し、保守性を高めます。
-   **責務の分離 (Separation of Concerns):**
    -   `main.py`の司令塔は、各機能の**マネージャークラス**（例: `AI_Manager`, `UI`）のみを直接インスタンス化し、管理します。
    -   特定のAIモデル（`Gemini`, `Ollama`）の選択や、UIの個別ウィンドウ（会話、設定）の管理といった下位の責務は、各マネージャークラスが担当します。これにより、司令塔は全体の連携のみに集中できます。

---

### 3. 最終決定: 移行の具体的な作業手順

以上の基本方針に基づき、EventBusを利用した新アーキテクチャへの移行は、以下の手順で進めることを最終方針とします。

#### ステップ1: `main.py`の準備 (司令塔の基盤整備)

1.  `from services.Event_Bus import EventBus` のように、`EventBus`クラスを`main.py`にインポートする。
2.  `myapp`クラスの`__init__`メソッドで、`self.event_bus = EventBus()` のようにインスタンスを一つ生成し、保持する。
3.  `UI`や`AI_Manager`など、各機能の**マネージャークラス**のインスタンスを生成する際に、引数として`self` (`myapp`インスタンス自身) を渡すように統一する。司令塔は、マネージャーが内部でどの下位クラス（例: `AI_geminiAPI`）を使うかを知る必要はない。
4.  `myapp`クラスに、中身がまだ空の `_setup_event_listeners(self)` メソッドを定義し、`__init__`の最後で呼び出しておく。

#### ステップ2: 各サービスの修正 (責務の分離とイベント発行)

1.  各**マネージャークラス** (例: `AI_Manager`) の`__init__`メソッドを修正し、引数で受け取った`context` (`myapp`インスタンス) を `self.context` のように保持する。
2.  **マネージャークラスの責務**として、設定情報を元に、使用する下位クラス（例: `AI_geminiAPI`や`AI_ollama`）のインスタンスを生成・管理する処理を実装する。
3.  サービス内で、他のサービスを直接呼び出している箇所を全て探し出し、`EventBus`によるイベントの発行に置き換える。

    ```python
    # 変更前: UIのメソッドを直接呼び出し
    self.context.ui.add_log("AIからの返信です")

    # 変更後: イベントを発行する
    self.context.event_bus.publish("ai_response_generated", "AIからの返信です")
    ```

#### ステップ3: `main.py`でのイベント購読 (集中管理の徹底)

1.  `_setup_event_listeners`メソッドの中に、アプリケーション全体のイベント購読（`subscribe`）処理を記述していく。
2.  このメソッドが、「どのイベントが、どのサービスの、どの関数を呼び出すか」という**唯一の対応表**となる。**原則として、アプリケーション内で `subscribe` メソッドが呼ばれるのは、この `_setup_event_listeners` 関数の中だけとする。**

    ```python
    # main.py の myapp._setup_event_listeners の中
    def _setup_event_listeners(self):
        """アプリケーション内のイベント購読をここで一元管理する"""

        # AIが応答を生成したら、UIの応答処理メソッドを呼び出す
        self.event_bus.subscribe("ai_response_generated", self.ui.handle_ai_response)
        
        # ユーザーがメッセージを送ったら、AIマネージャーの処理メソッドを呼び出す
        self.event_bus.subscribe("user_message_sent", self.ai_manager.handle_user_message)
        
        # ... 他のすべてのイベント配線をここに記述 ...
    ```

#### ステップ4: 動作確認

1.  一連の修正が完了したら、アプリケーションを起動する。
2.  メッセージの送受信など、イベントが関連する一連の動作が、リファクタリング前と同様に問題なく実行されることを確認する。
