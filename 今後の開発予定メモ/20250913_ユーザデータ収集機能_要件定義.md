# ユーザーデータ収集と活用機能に関する要件定義

**更新日:** 2025年9月13日
**担当:** 開発担当者, AIアシスタント (Gemini)

---

## 1. 目的

ユーザーのPC利用状況を記録し、そのデータをLLMとの会話に活用することで、よりユーザー個人に寄り添った、パーソナルな対話体験を実現する。具体的には、ユーザーの活動を振り返る機会の提供や、状況に応じた能動的な働きかけを目指す。

## 2. 収集する情報

以下の3項目を1つのデータセットとして記録する。

- **タイムスタンプ:** データが記録された日時 (ISO 8601形式)
- **アクティブウィンドウタイトル:** ユーザーがその時に使用していたアプリケーションのウィンドウタイトル
- **再生メディア情報:** 再生中の音楽や動画のタイトルとアーティスト名 (取得可能な場合)

### ログの形式

データは後続処理のしやすさを考慮し、**JSON形式**で記録する。

## 3. データ収集・管理方式

### 収集方式：ハイブリッド方式
PCへの負荷とデータ精度のバランスを考慮し、以下の2つの方式を組み合わせた**ハイブリッド方式**を採用する。

1.  **イベント駆動収集:** ユーザーがアクティブウィンドウを切り替えたタイミングで記録する。
2.  **定周期収集:** **5分ごと**に、現在の状況を強制的に記録する。

### 管理方式：ログローテーション
単一ファイルの肥大化を防ぎ、管理を容易にするため、**日次（daily）**でログファイルを分割するログローテーションを導入する。

- **ファイル名形式:** `activity_log_YYYY-MM-DD.json`
- **保存場所:** `user_data/logs/` （設定で変更可能とする）

## 4. 想定される活用案

- **休息の提案:** 長時間、同じ作業が続いている場合に休憩を提案する。
- **苦戦している作業のヒアリング:** 特定の作業とWebブラウザとの間でウィンドウの行き来が頻繁な場合に、問題の整理を助けるか働きかける。
- **アクティビティの要約:** 「今日の午前中は何をしていた？」といった問いに、ログを基に回答する。

## 5. (補足) 収集方式の比較検討

| 方式 | 利点 | 欠点 | 採用判断 |
| :--- | :--- | :--- | :--- |
| **1. 定期間隔** | - 実装が容易<br>- データ量が予測しやすい | - 短時間の作業を取りこぼす<br>- 作業の開始/終了時刻の精度が低い | × 活用案の実現には精度不足 |
| **2. イベント駆動** | - 作業遷移を正確に把握<br>- 実質的な行動のみ記録 | - 長時間作業の記録が抜ける | △ これだけでは不十分 |
| **3. ハイブリッド** | - 両方式の利点を享受<br>- 高精度な分析が可能 | - 実装がやや複雑になる | ○ 採用 |

## 6. その他の考慮事項

- **プライバシー:** ウィンドウタイトルに機密情報が含まれる可能性は否定できないが、ログは外部に送信されず、ローカルで完結するため、リスクは限定的と判断。将来的にマスキング機能の追加を検討する可能性はある。
- **PC負荷:** ウィンドウの常時監視は、OSへの問い合わせ処理自体はCPU負荷が極めて軽微である。さらに、監視処理をバックグラウンドスレッドで実行することで、アプリケーション全体のパフォーマンスへの影響を最小限に抑える設計とする。

## 7. 将来的な発展項目と展望

本要件定義は機能実装の第一歩であり、長期的には以下の発展項目が考えられる。

### 発展的な機能案

- **活動データの可視化:**
  - 収集したログを基に「日次・週次の活動サマリー」や「アプリケーション利用時間のグラフ」などを生成し、ユーザーが自身の活動を視覚的に振り返れるUIを提供する。これにより、ユーザーは自身の生産性や習慣を客観的に把握できるようになる。

- **自己診断セーフティネット:**
  - アプリケーションが自身のCPUやメモリ使用量を簡易的に監視する機能を搭載する。万が一、バックグラウンドでの監視処理が予期せぬ負荷をかけた場合に、自動で監視間隔を調整したり、ユーザーに警告したりすることで、安定性を向上させる。

### 現行計画の弱点と今後の改善の方向性

- **データの「質」と「文脈」の深化:**
  - **課題:** 現行の「ウィンドウタイトル」のみの収集では、ユーザーの真の意図や文脈を正確に把握するには限界がある（例: 同じブラウザでも閲覧サイトによって活動内容は異なる）。
  - **今後の展望:** よりリッチな文脈を理解するため、将来的にはブラウザのアクティブなタブURLや、エディタで開いているファイル名などを取得する拡張を検討する。ただし、これはプライバシーとの強いトレードオフになるため、実装にはユーザーの明確な同意と、マスキング機能の強化が前提となる。

- **LLM活用に向けたデータの前処理:**
  - **課題:** 生の時系列ログは、そのままではLLM（特にRAG）が効率的に情報を引き出すのに適していない可能性がある。
  - **今後の展望:** ログをLLMが扱いやすい形式に前処理する中間ステップを導入する。例えば、1時間ごとに「作業サマリー」を生成して別途保存しておく、あるいは特定のイベント（コーディング開始、資料作成終了など）をタグ付けしておくなどの手法が考えられる。

- **ユーザー体験（UX）のパーソナライズ:**
  - **課題:** アプリケーションからの能動的な働きかけ（休息の提案など）は、ユーザーによっては「おせっかい」と感じられ、集中を阻害する可能性がある。
  - **今後の展望:** 提案の通知レベルをユーザーが細かく制御できる設定画面を設ける。「休息の提案は有効にするが、作業のヒアリングは無効にする」といったように、ユーザーがアプリケーションとの快適な距離感を選択できるようにすることが、長期的な利用には不可欠となる。
