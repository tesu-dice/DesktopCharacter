# LLMリクエストキュー & ユーザープロファイル 設計方針

**作成日:** 2026年06月05日  
**対象プロジェクト:** DesktopCharacter

---

## 1. 背景と課題

### 1.1 現状の問題

LLMへのリクエストが複数の発生源から無秩序に送られることで、以下の問題が起きていた。

- ユーザー入力・自発的会話・サマリー生成が同時に走りタイムアウトが発生する
- 23:55に時間サマリーと日サマリーが重複してエラーになる
- どの生成結果がどのリクエストへの返答かをEventBusの受信側が識別できない

### 1.2 解決方針

LLMへのリクエストを**優先度付きキュー**に一本化し、ワーカースレッドが1件ずつ処理する。
リクエストには**request_id**と**response_event**を持たせることで、返答の識別を可能にする。

---

## 2. LLMリクエストキューの設計

### 2.1 優先度の定義

| 優先度 | 種別 | 説明 |
|--------|------|------|
| 1（最高）| ユーザー入力への応答 | ユーザーが直接入力したメッセージへの返答 |
| 2 | 自発的会話 | アプリからの能動的な話しかけ |
| 3 | サマリー生成 | 時間別・日別のアクティビティサマリー |
| 4（最低）| プロフィール更新 | ユーザープロファイルの自動更新 |

ユーザーが入力した瞬間に優先度1のリクエストがキューに入ると、優先度3・4の処理は後回しになる。

### 2.2 リクエストのデータ構造

```json
{
  "request_id": "uuid-xxxx",
  "priority": 1,
  "payload": "LLMへ渡すinput_contentsのリスト",
  "response_event": "OnUserResponse"
}
```

- `request_id`: UUIDで生成する一意な識別子
- `priority`: 数値が小さいほど優先度が高い
- `payload`: AI_mainの各response関数に渡すinput_contents
- `response_event`: 生成完了時に発行するEventBus上のイベント名

### 2.3 処理の流れ

```
各モジュール
  ↓ bus.publish("Req_LLM", request_dict)
EventBus
  ↓
LLMQueue（優先度付きキュー）
  ↓ ワーカースレッドが1件ずつ取り出す
AI_main.response()
  ↓ 生成完了
bus.publish(response_event, {"request_id": "uuid-xxxx", "text": "生成結果"})
  ↓
各モジュールが自分のrequest_idを確認して受け取る
```

### 2.4 返答の識別方法

**イベント名で「誰への返答か」を区別し、request_idで「どのリクエストへの返答か」を確認する。**

```
ユーザー入力を送った側  → response_event: "OnUserResponse"    を購読
サマリー生成を頼んだ側  → response_event: "OnSummaryDone"     を購読
プロフィール更新を頼んだ側 → response_event: "OnProfileDone" を購読
```

同じイベント名が複数飛んでくる場合（例：連続したユーザー入力）は、
`request_id`を照合して自分のリクエストへの返答か確認する。

### 2.5 既存アーキテクチャへの組み込み

既存の`EventBus`と`subscribe_workflow`の仕組みをそのまま活かす。
`AI_main`の各関数（`response()`・`response_withRAG()`・`response_onetime()`）は
すべてキュー経由に統一し、呼び出し側は「投げたら完了イベントを待つ」スタイルに統一する。

## 3. ユーザープロファイルの設計

### 3.1 ファイル構成

プロファイルを2ファイルに分けて管理する。

| ファイル | 内容 | 更新頻度 |
|----------|------|----------|
| `user_profile_stable.json` | 職業・よく使うツール・活動時間帯など変化が少ない情報 | 変化があったときのみ上書き |
| `user_profile_recent.json` | 直近1〜2週間の傾向・最近取り組んでいること | 毎回の更新で丸ごと上書き |

データが膨らみ続けないよう、`user_profile_recent.json`は**常に上書き**とする。
過去の日サマリーファイル自体はそのまま保持されるため、必要なら遡ることもできる。

### 3.2 プロファイルの内容（例）

```json
{
  "last_updated": "2026-06-01",
  "stable": {
    "occupation": "プログラマー",
    "main_tools": ["VSCode", "Python", "Git"],
    "active_hours": "夜型（21時〜深夜）"
  },
  "recent": {
    "current_topics": ["DesktopCharacterの開発", "EventBusの設計"],
    "recent_media": ["ジャズ系BGM"],
    "work_pattern": "長時間の集中作業が多い"
  }
}
```

### 3.3 init_promptへの組み込み

アプリ起動時にプロファイルファイルを読み込み、`init_prompt`の末尾に追加する。

```
init_prompt = [
  キャラクター設定,
  立ち絵一覧,
  ユーザープロファイル（stable + recent）  ← 追加
]
```

毎回の会話コストへの影響はプロファイルの文字数分のみで、数百トークン程度に収まる。

## 4. プロファイル更新のタイミング

### 4.1 基本方針

**アプリ起動時に前回更新日時を確認し、N日以上経過していたらバックグラウンドで更新する。**

定刻（23:55など）での更新は他の処理との競合を生むため採用しない。
起動時チェックにすることで、ユーザーが使っている最中にLLMリクエストが走らない。

```
アプリ起動
  ↓
last_updated を確認
  ↓
N日以上経過していたら？
  ↓ Yes
優先度4でキューにプロファイル更新リクエストを積む
  ↓
ワーカーがアイドル時に処理
  ↓ 完了
last_updated を更新・プロファイルファイルを上書き保存
```

失敗した場合は次回起動時に再度チェックされるため、特別なリトライ処理は不要。

### 4.2 更新頻度の推奨

| 頻度 | 特徴 |
|------|------|
| 毎日 | プロファイルの鮮度は高いが、起動のたびに処理が走る |
| **週1回（推奨）** | 体感への影響が少なく、陳腐化も許容範囲内 |
| 手動のみ | コスト最小だが自動更新の恩恵がない |

週1回を基本とし、設定UIからユーザーが頻度を変更できるようにするとよい。

### 4.3 更新処理の入力データ

LLMへの入力は**直近7日分の日サマリーのみ**に絞る。
過去の全サマリーを渡す必要はなく、トークン数が予測可能な範囲に収まる。

プロンプトイメージ：
```
以下はユーザーの直近7日間の活動サマリーです。
この情報からユーザーの傾向・特徴を20項目以内の箇条書きで抽出してください。

[日サマリー × 7日分]
```

更新処理はOllamaで行うことを推奨する（Gemini無料枠の消費を避けるため）。
設定でAIサービスを切り替えられる既存の仕組みをそのまま活用できる。

## 5. サマリー生成との関係

サマリー生成（時間別・日別）もプロファイル更新と同様に、
定刻ではなく**起動時チェックまたはキュー経由**に切り替えることで、
競合問題を根本から解消できる。

```
現状（問題あり）:
  23:55 → 時間サマリー生成 と 日サマリー生成 が同時に走る → タイムアウト

改善後:
  起動時 → 前回未生成分をキューに積む（優先度3）
  ワーカーが会話の合間に順番に処理 → 競合なし
```

---

## 6. 実装優先順位

1. **LLMリクエストキューの実装**（最優先）
   - 優先度付きキューとワーカースレッドの作成
   - request_id / response_event の仕組み
   - 既存のresponse関数をキュー経由に統一

2. **サマリー生成の起動時チェックへの移行**
   - 定刻トリガーを廃止し、起動時チェックに変更
   - キューに優先度3で積む

3. **ユーザープロファイル管理の実装**
   - user_profile_stable.json / user_profile_recent.json の作成
   - 起動時チェック・週次更新ロジック
   - init_promptへの組み込み
