# [Feature] ReAct型エージェントによるObsidian連携基盤の開発とツール実行中継システムの実装

## 1. 概要
デスクトップ常駐型AIとしての「思考の深さ」と「キャラクター性」を両立するため、ReActパターンの推論エンジンを実装する。RAGフォルダの参照(Obsidian内の知識探索やWeb情報の取得)を自律的に行いつつ、ツール実行中の待ち時間をキャラクターの「中継セリフ」で埋めることで、ユーザー体験を向上させる。

また、将来的な拡張として「Multi-Agent型」での応答(自由なキャラ設定と競合する場合あり)も強力な一つの「ツール」として組み込める柔軟な設計を目指す。

## 2. アーキテクチャ方針
* **ReAct推論**: 「Thought（思考）→ Action（ツール実行）→ Observation（結果確認）→ Final Answer（回答）」のサイクルを回す。
* **中継セリフ機能**: ツール実行中、沈黙を避けるためにキャラクター設定に基づいた進捗報告セリフ（中継セリフ）を生成・表示する。
* **Multi-Agentへの布石**: 複雑なタスクを「特定の専門家エージェントに依頼するツール」として定義し、現在のTool枠組みを維持したまま統合可能にする。

## 3. ReAct用システムプロンプト（実装リファレンス）
```text
あなたはユーザーをサポートするAIパートナー（キャラ名：[設定]）です。
以下の手順で思考し、問題を解決してください。

1. Thought: 必要な情報や次のステップを論理的に整理する。
2. Action: ツールが必要な場合、指定のJSON形式で呼び出す。
3. Observation: ツールの結果を客観的に評価する。
4. Final Answer: 全ての情報を統合し、最後に【キャラクターらしい口調】で回答する。

【ツール呼び出し形式】
ツール使用時は、以下のJSONのみを出力してください。
{"tool": "関数名", "args": {"引数": "値"}}

・ThoughtやActionは簡潔な日本語で行う。
・Final Answerのみ、設定されたキャラクターとして振る舞うこと。
```

## 4. 実装ステップ（チェックリスト）

### Phase 1: ReActコア・ループの実装
- [ ] LLMへのツール定義（JSON Schema）送付処理。
- [ ] ツール実行と結果フィードバックの `while` ループ構築。
- [ ] キャラクター中継セリフの表示ロジック（ツール呼び出し検知時に実行）。

### Phase 2: Obsidian階層探索ツールの実装
再帰処理を使って枝分かれしつつも調べられたら最高。とりあえずはその場その場でチェックしつつ一本道を探索していくイメージ  
- [ ] `get_relevant_folders`: 質問から探索対象フォルダを絞り込む。
- [ ] `list_files_in_folder`: フォルダ内のファイル名一覧を取得し、LLMに選択させる。
- [ ] `read_obsidian_note`: 特定されたMarkdownファイルの内容読み込み。

### Phase 3: Web連携・Multi-Agent拡張
- [ ] `web_content_fetcher`: WebClipフォルダ内のURL先を取得・要約するツール。
- [ ] **(Future)** `multi_agent_delegate`: 別エージェントをツールとして呼び出すインターフェースの設計。

## 5. 期待される動作フロー（中継セリフのイメージ）
1. **ユーザー**: 「PCが重い原因を過去のメモから探して」
2. **AI (Action)**: `search_obsidian` を呼び出し。
3. **UI (キャラクター中継セリフ)**: 「ちょっと待ってて、あなたのメモの中から原因になりそうなのを探してくるから！（探索中...）」
4. **AI (Final Answer)**: 「お待たせ！去年のメモに『スタートアップアプリを減らす』って書いてあったよ。試してみる？」

## 6. 備考
- キャラクターの中継セリフは、ツール名ごとにランダムなバリエーションを持たせることで「生きている感」を演出する。
- Multi-Agent化の際は、特定エージェントの出力を Observation として受け取ることで、現在のReActフローを維持する。
