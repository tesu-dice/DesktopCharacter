# ユーザーデータ収集と活用機能に関する要件定義

---

## 1. 目的

ユーザーのPC利用状況を記録し、そのデータをLLMとの会話に活用することで、よりユーザー個人に寄り添った、パーソナルな対話体験を実現する。具体的には、ユーザーの活動を振り返る機会の提供や、状況に応じた能動的な働きかけを目指す。

## 2. データ収集

以下の5項目を1つのデータセット(json)として記録する。  
記録周期は5分ごととし、収集の際にの自発的会話と重複しないよう自発的会話はログを残した後に確認する。  
周期的な記録を参照して時間ごとおよび1日ごとにそれぞれまとめた5行程度の文章にLLMを用いてまとめる。
これは11:55に記録を作成したのちにそのまま11時のサマリーテキストを作成する。
一日の作業内容に関してはいったん23:55の記録後に連続して作成するとするが、ユーザの指示によってのみ作成する形式またはGoogleCalendarとの連携機能による日記作成機能と統合することを検討する。
LLM側は、必要に応じてこれらのデータをRAGとして参照する。


| 項目 | 形式 | 説明 |
| :--- | :--- | :--- |
| タイムスタンプ | str | データが記録された日時 |
| アクティブウィンドウタイトル | str | ユーザーがその時に使用していたアプリケーションのウィンドウタイトル | |
| 再生メディア情報 | str | 再生中の音楽や動画のタイトルとアーティスト名 (取得可能な場合) |

---
データ形式(YYYYMMDD.json)
```json
{
  "logs":{
    {
      "time":"TT:MM:SS",
      "window":"アクティブなウィンドウの名前",
      "media":"再生中のメディア"
    },
    {
      "time":"TT:MM:SS",
      "window":"アクティブなウィンドウの名前",
      "media":"再生中のメディア"
    }
  },
  "hourlogs": {
    {
      "time": "TT",
      "summary": "ユーザはプログラミングをしていた"
    },
    {
      "time": "TT",
      "summary": "ユーザはブラウザで休憩していた。"
    }
  },
  "daylogs": {
    "summary":"ユーザはおよそn時間にわたりプログラミングを行っていた。"
  }
}

```


## 3. 収集データの活用

- **休息の提案:** 長時間、同じ作業が続いている場合に休憩を提案する。
- **苦戦している作業のヒアリング:** 特定の作業とWebブラウザとの間でウィンドウの行き来が頻繁な場合に、問題の整理を助けるか働きかける。
- **アクティビティの要約:** 「今日の午前中は何をしていた？」といった問いに、ログを基に回答する。




- **活動データの可視化:**
  - 収集したログを基に「日次・週次の活動サマリー」や「アプリケーション利用時間のグラフ」などを生成し、ユーザーが自身の活動を視覚的に振り返れるUIを提供する。これにより、ユーザーは自身の生産性や習慣を客観的に把握できるようになる。

- **自己診断セーフティネット:**
  - アプリケーションが自身のCPUやメモリ使用量を簡易的に監視する機能を搭載する。万が一、バックグラウンドでの監視処理が予期せぬ負荷をかけた場合に、自動で監視間隔を調整したり、ユーザーに警告したりすることで、安定性を向上させる。

### 現行計画の弱点と今後の改善の方向性

- **データの「質」と「文脈」の深化:**
  - **課題:**  
  現行の「ウィンドウタイトル」のみの収集では、ユーザーの真の意図や文脈を正確に把握するには限界がある（例: 同じブラウザでも閲覧サイトによって活動内容は異なる）。
  - **今後の展望:**  
  よりリッチな文脈を理解するため、将来的にはブラウザのアクティブなタブURLや、エディタで開いているファイル名などを取得する拡張を検討する。ただし、これはプライバシーとの強いトレードオフになるため、実装にはユーザーの明確な同意と、マスキング機能の強化が前提となる。

- **LLM活用に向けたデータの前処理:**
  - **課題:**  
  生の時系列ログは、そのままではLLM（特にRAG）が効率的に情報を引き出すのに適していない可能性がある。
  - **今後の展望:**  
  ログをLLMが扱いやすい形式に前処理する中間ステップを導入する。例えば、1時間ごとに「作業サマリー」を生成して別途保存しておく、あるいは特定のイベント（コーディング開始、資料作成終了など）をタグ付けしておくなどの手法が考えられる。

- **ユーザー体験（UX）のパーソナライズ:**
  - **課題:**  
  アプリケーションからの能動的な働きかけ（休息の提案など）は、ユーザーによっては「おせっかい」と感じられ、集中を阻害する可能性がある。
  - **今後の展望:**  
  提案の通知レベルをユーザーが細かく制御できる設定画面を設ける。「休息の提案は有効にするが、作業のヒアリングは無効にする」といったように、ユーザーがアプリケーションとの快適な距離感を選択できるようにすることが、長期的な利用には不可欠となる。





## その他の考慮事項
- **ユーザの意思の尊重:**  
ユーザによっては記録機能を希望しない可能性があるため設定項目によってこれらの機能を利用するかを選択可能とする。
- **プライバシー:** ウィンドウタイトルに機密情報が含まれる可能性は否定できないが、ログは外部に送信されず、ローカルで完結するため、リスクは限定的と判断。将来的にマスキング機能の追加を検討する可能性はある。
- **PC負荷:** ウィンドウの常時監視は、OSへの問い合わせ処理自体はCPU負荷が極めて軽微である。さらに、監視処理をバックグラウンドスレッドで実行することで、アプリケーション全体のパフォーマンスへの影響を最小限に抑える設計とする。
